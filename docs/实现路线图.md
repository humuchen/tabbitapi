> 配合 [逆向流程与协议.md](逆向流程与协议.md) 阅读。本文规划项目结构、技术栈、实现步骤与 OpenAI 兼容映射。

---

## 1. 技术栈建议

**Node.js + TypeScript**（首选）

理由：
- 原始逆向代码是 JS，签名/SSE 逻辑可 1:1 移植，零翻译误差
- `crypto.subtle` / `TextEncoder` / `fetch` / `ReadableStream` 在 Node 18+ 全部原生可用，与浏览器一致
- SSE 流式转发用原生流，性能好

依赖（极简）：
- `fastify` 或原生 `http` —— HTTP 服务
- 无需其他，签名/SSE 全用内置 `crypto`、`stream`

（若你更想要 Python，可改用 Python 3.11 + FastAPI + httpx + cryptography；但 HMAC/SSE 需手写，且 sha256/hex 细节易出错。下面以 Node 为准。）

## 3. 实现步骤（按依赖顺序）

### Step 0 · 探测验证（最先做）

写 `scripts/probe.ts`，用真实 Cookie 依次打通三件事，**确认逆向文档准确**：

1. `GET /chat/sign-key` → 拿动态 key（验证 Cookie 有效）
2. `GET /proxy/v1/model_config/models?a=0&scene=chat` → 拿模型列表（验证签名通过）
3. `POST /api/v2/chat/completion` → 最小 body，打印原始 SSE 流（验证聊天通）

每步存请求/响应到 `logs/probe-<n>.json`。根据结果回填协议文档的「待验证清单」。

**只有 Step 0 跑通，后面才有意义。**

### Step 1 · 认证模块 `src/tabbit/auth.ts`

实现：
- `uuid()` —— `crypto.randomUUID()`
- `withTraceId(headers)` —— 注入 `trace-id`
- `sha256Hex(text)` —— `crypto.createHash('sha256').update(text).digest('hex')`
- `hmacSha256Hex(key, text)` —— `crypto.createHmac('sha256', key).update(text).digest('hex')`
- `signHeaders(body, key)` —— 返回 `{x-timestamp, x-nonce, x-signature}`
  - `ts = Date.now().toString()`
  - `sig = uuid()`（放 x-signature）
  - `msg = ${ts}.${sig}.${sha256Hex(body)}`
  - `nonce = hmacSha256Hex(key, msg)`（放 x-nonce）
- `refreshSignKey(cookie)` —— `GET /chat/sign-key`，返回新 key 或 null
- `SignKeyStore` —— 进程内缓存当前 key，支持 499 后刷新

伪代码：
```ts
export async function signHeaders(body: string, key: string) {
  const ts = String(Date.now());
  const sig = randomUUID();
  const msg = `${ts}.${sig}.${sha256Hex(body)}`;
  const nonce = hmacSha256Hex(key, msg);
  return { 'x-timestamp': ts, 'x-nonce': nonce, 'x-signature': sig };
}
```

### Step 2 · SSE 解析器 `src/tabbit/sse.ts`

移植 `x()` 函数：输入 `ReadableStream<Uint8Array>`，输出异步迭代的事件 `{event, data, id}`。

```ts
export async function* parseSSE(stream: ReadableStream<Uint8Array>) {
  const reader = stream.getReader();
  const decoder = new TextDecoder();
  let buffer = '', event = '', data = '', id = '';
  for (;;) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n');
    buffer = lines.pop() || '';
    for (const line of lines) {
      const t = line.trim();
      if (t === '') { if (data) yield { event: event || 'message_chunk', data, id }; event=''; data=''; id=''; continue; }
      if (t.startsWith('event: ')) event = t.slice(7).trim();
      else if (t.startsWith('data: ')) data = data ? data+'\n'+t.slice(6) : t.slice(6);
      else if (t.startsWith('id: ')) id = t.slice(4).trim();
    }
  }
}
```

### Step 4 · 格式转换 `src/openai/transform.ts`

**OpenAI 请求 → Tabbit 请求**：

| OpenAI 字段 | Tabbit 字段 |
|-------------|-------------|
| `messages[].role/content` | 拼成 `content`（首期单轮；多轮可拼历史进 content 或逐轮回放，见下） |
| `model` | `selected_model`（需先查模型列表确认 ID 格式） |
| `stream` | 固定 `stream_mode:"sse"` |
| `temperature` / `max_tokens` | 放 `metadatas`（待实测字段名） |

> **多轮对话处理**：Tabbit 是会话制（`chat_session_id`），OpenAI 是无状态 messages 数组。两种策略：
> - **简单**：把历史 messages 拼成一段 content 发送（丢上下文管理能力）
> - **会话映射**：用 `client_turn_id` + `chat_session_id` 维护会话，首轮新建，后续轮复用。Tabbit 的 `message_finish` 事件可能回传 `chat_session_id`，据此缓存。
>
> 首期实现「简单」策略，后续升级「会话映射」（可按 OpenAI 请求里的自定义字段 `chat_session_id` 透传）。

**Tabbit 事件 → OpenAI SSE**：

| Tabbit 事件 | OpenAI 输出 |
|-------------|-------------|
| `message_start` | `{"choices":[{"delta":{"role":"assistant"},"index":0}]}` 首块 |
| `message_chunk` (data.content) | `{"choices":[{"delta":{"content":"..."},"index":0}]}` |
| `message_finish` / `finish` | `{"choices":[{"delta":{},"finish_reason":"stop"}]}` + `data: [DONE]` |
| `usage` | `{"usage":{...}}` |
| `error` | 流内 error 或提前结束 + 非 200 |

非流式：聚合所有 `message_chunk.content`，一次性返回 OpenAI `ChatCompletion` JSON。

## 6. 注意事项

1. **先 P0 再写代码**：逆向文档里标了 7 个「待验证」点，必须先实测确认 `selected_model` 字段名、session 可空性、chunk 结构，否则后面全白写。
2. **签名 header 命名**：先用逆向结果（`x-nonce` 放 HMAC、`x-signature` 放 UUID）。若 P0 探测签名失败，第一步尝试交换两者；仍失败则重新抓 Tabbit 浏览器实际请求的 header 顺序核对。
3. **GET 请求签名**：body 为空字符串，`sha256("") = e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`。确认服务端接受。
4. **Cookie 安全**：Cookie 等同账号控制权，`.env` 加入 `.gitignore`，勿提交。
5. **合规**：仅供个人学习研究，不要商用、不要高并发滥用、不要二次分发账号。
