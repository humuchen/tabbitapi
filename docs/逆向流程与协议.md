## 1. 架构总览

Tabbit 是 Chromium 套壳浏览器，但 AI 对话逻辑全在 **Next.js 前端**里，运行在浏览器侧边栏面板。原生侧（`Tabbit.dll`）只负责 Skills 下载、桌面宠物等。

```
Tabbit 浏览器侧边栏 (Next.js 前端)
   │
   │  ① 模型列表  GET /proxy/v1/model_config/models
   │  ② 聊天     POST /api/v2/chat/completion  (SSE)
   │  ③ 停止     POST /proxy/v0/chat/stop/
   │  ④ Agent    WSS /api/agent/v2/ws   (浏览器自动化，本项目暂不实现)
   │
   ▼
https://web.tabbit.ai   (Tabbit 自有后端，转发到真实 LLM)
```

**本项目只复用 ①②③**，把 HTTP 路径做成 OpenAI 兼容 API。Agent WebSocket 路径（浏览器自动化）不在首期范围。

后端域名映射（dll 中可见）：

| 域名 | 用途 |
|------|------|
| `web.tabbit.ai` | 主 API（聊天/会话/配额/签名） |
| `cdn.tabbit.ai` | 前端静态资源 |
| `skills.tabbit-ai.com` | 第一方 Skills 二进制下载 |
| `plaza.tabai-test.meituan.com` | 测试环境（美团系） |

## 2. 认证四要素

### 2.1 Cookie（登录态）

所有请求 `credentials:"include"`，依赖 Cookie 维持会话。需要从已登录的 Tabbit 浏览器中导出 `web.tabbit.ai` 域下的 Cookie（主要是 session/token 类）。

**导出方法**：打开 Tabbit → DevTools → Application → Cookies → `https://web.tabbit.ai` → 全选复制；或用浏览器扩展导出 Cookie 字符串。配置到本项目的 `TABBIT_COOKIE` 环境变量。

### 2.2 trace-id header

每个请求生成一个 UUID v4 放入 `trace-id` header。代码（67174 模块）：

```js
function s(e){
  let t = e instanceof Headers ? Object.fromEntries(e.entries()) : e||{};
  if(!Object.keys(t).some(e=>"trace-id"===e.toLowerCase()&&!!t[e]))
    t["trace-id"] = a();   // a() = crypto.randomUUID()
  return t;
}
```

### 2.3 HMAC 签名（核心）

对 **请求 body 字符串**做签名，生成三个 header。原文（60442 模块）：

```js
let l = "f8d0e6a73f8d4b1a9c3d2e1f9a4b7c6d", u = null;

// bytes → hex
function h(e){
  return Array.from(new Uint8Array(e)).map(e=>e.toString(16).padStart(2,"0")).join("");
}

// 导入 HMAC key
async function p(e){
  if(u && u.raw===e) return u.cryptoKey;
  let t = await crypto.subtle.importKey(
    "raw",
    new TextEncoder().encode(e),
    {name:"HMAC", hash:"SHA-256"}, false, ["sign"]
  );
  u = {raw:e, cryptoKey:t};
  return t;
}

// 签名主函数，e = body 字符串
async function y(e){
  if(!l) return {};
  let t = String(Date.now()),                          // x-timestamp (毫秒)
      r = crypto.randomUUID(),                          // 随机 UUID
      s = new TextEncoder(),
      n = h(await crypto.subtle.digest("SHA-256", s.encode(e))),  // sha256(body) hex
      i = `${t}.${r}.${n}`,                             // 待签名消息
      o = await p(l);                                   // HMAC key
  return {
    "x-timestamp": t,
    "x-nonce":     h(await crypto.subtle.sign("HMAC", o, s.encode(i))),  // HMAC hex
    "x-signature": r                                    // UUID
  };
}
```

**注意命名反直觉**（混淆所致，以代码行为为准）：

| header | 值 |
|--------|----|
| `x-timestamp` | `Date.now()` 毫秒数字符串 |
| `x-signature` | 随机 UUID v4 |
| `x-nonce` | `HMAC-SHA256(KEY, "${timestamp}.${uuid}.${sha256(body)}")` 的 hex |

其中 `KEY` 默认硬编码 `f8d0e6a73f8d4b1a9c3d2e1f9a4b7c6d`。

### 2.4 sign-key 刷新（499 重试）

服务端可动态更换 key。客户端在响应 **HTTP 499** 时刷新 key 并重试一次。原文：

```js
async function f(){
  try{
    let e = await fetch("/chat/sign-key");          // GET https://web.tabbit.ai/chat/sign-key
    if(!e.ok) return console.warn("[refreshSignKey] fetch failed:", e.status), false;
    let t = (await e.text()).trim();                 // 纯文本新 key
    if(t && t!==l){ l = t; u = null; return true; }  // 更新 key，清空缓存的 cryptoKey
  }catch(e){ console.warn("[refreshSignKey] error:", e); }
  return false;
}

// 请求器中：
fetch(url, opts)
  .then(async e => 499===e.status && await f() ? fetch(url, opts) : e)  // 499 则刷新+重试一次
  .then(...)
```

**实现要点**：
- 初始用硬编码 key；遇 499 调 `GET /chat/sign-key`（带 Cookie）拿新 key；
- 新 key 在进程内缓存，后续请求都用它；
- key 失效后再次 499 再刷新。

### 2.5 MOA 证书标记（影响模型列表）

前端通过原生扩展命令 `CHECK_MOA_CERTIFICATE`（`extensionId: c.c3`）判断是否内部/MOA 用户，结果影响模型列表接口的 `a` 参数：

```js
async getSupportedModels(e="chat"){
  let t = new URLSearchParams({ a: await this.checkMoaCertificate() ? "1" : "0", scene: e });
  return this.get(`/proxy/v1/model_config/models?${t.toString()}`);
}
```

外部用户一律 `a=0`。本项目固定 `a=0` 即可（除非你有 MOA 证书环境）。

### 2.6 `unique-uuid` header（Pro 会员自证机制，重要）

Tabbit 的 Pro 会员有一条**非付费路径**：把 Tabbit 设为 Windows 默认浏览器 → 解锁 Pro。这条路径不调任何「领取」API，而是靠 `unique-uuid` 请求头**客户端自证**。

#### 生成算法（模块 1711）

```js
let s = {
  markerPos: 5,                          // 标记位在 UUID 第 5 位（0-indexed）
  defaultBrowserMarker: "1",             // 已设默认浏览器 → 该位为 "1"
  timestampPositions: [2,7,11,14,18,21,25,28]  // 这些位编码当前时间 hex（防重放）
};

function n(isDefault, t=s){
  let hex = "0123456789abcdef",
      others = hex.replace(t.defaultBrowserMarker, ""),   // 去掉 "1" 的 hex 集
      ts = Math.floor(Date.now()/1e3).toString(16).padStart(8,"0").slice(-8),
      tsMap = new Map();
  t.timestampPositions.forEach((pos,i) => tsMap.set(pos, ts[i]));
  let out = "";
  for(let i=0;i<32;i++){
    if(i === t.markerPos)
      out += isDefault ? t.defaultBrowserMarker : others[Math.floor(Math.random()*others.length)];
    else if(tsMap.has(i))
      out += tsMap.get(i);
    else
      out += hex[Math.floor(Math.random()*16)];
  }
  return [out.slice(0,8),out.slice(8,12),out.slice(12,16),out.slice(16,20),out.slice(20,32)].join("-");
}
```

UUID 结构：`xxxxxx1x-xxxx-xxxx-xxxx-xxxxxxxxxxxx`（第 5 位 = 默认浏览器标记，`1`=已设）。

#### 检测与上报（模块 38363）

```js
// 调 Tabbit 注入的原生扩展 API 检测是否系统默认浏览器
async function d(){
  try{
    let e = chrome?.tabInstance;
    if(e && typeof e.isDefaultBrowser === "function")
      return new Promise(t => { e.isDefaultBrowser(r => t(r)) });
    return false;
  }catch{ return false; }
}

// 组装头：{ "unique-uuid": n(isDefault) }
async function u(){
  let e = await d();
  return { "unique-uuid": n(e) };
}

// 合并指纹头：{ "x-req-ctx": btoa(tabbitVersion), "unique-uuid": <带标记的UUID> }
async function m(){
  let [a,b] = await Promise.all([o(), u()]);  // o() = {x-req-ctx: base64(version)}
  return {...a, ...b};
}
```

`m()` / `HH` 返回的 `{x-req-ctx, unique-uuid}` 被 spread 进每次 API 请求的 headers。

#### Pro 解锁流程

1. **触发**：用户在升级页 / 设置页 / 聊天报错提示点「设为默认浏览器」
2. **前端调用**：`chrome.tabInstance.setAsDefaultBrowser(callback)`
3. **原生执行**：`Tabbit.dll` 的 Chromium `DefaultBrowserManager` 写 Windows 注册表 `UserChoice`（`HKCU\Software\Microsoft\Windows\Shell\Associations\UrlAssociations\{http,https}\UserChoice`），把 Tabbit 设为系统默认浏览器
4. **确认**：5 秒后前端再调 `isDefaultBrowser` 验证成功
5. **UI 即时生效**：升级页代码 `v(true); g("pro")` —— 立即显示 Pro 方案，**不调任何后端 API**
6. **后端解锁**：后续所有请求的 `unique-uuid` 标记位变 `1`，后端检测到 → 不再返回 492/493，Pro 功能生效

升级页原文（`page-71a16bbbddcd6588.js`）：
```js
let C = useCallback(async () => {
  try{
    let e = chrome?.tabInstance;
    if(typeof e?.setAsDefaultBrowser !== "function" ||
       !await new Promise(t => { e.setAsDefaultBrowser?.(r => t(r)) })) return;
    _.current = setTimeout(async () => {
      if(typeof e.isDefaultBrowser !== "function" ||
         await new Promise(t => { e.isDefaultBrowser?.(r => t(r)) })) {
        v(true);        // setIsDefaultBrowser(true)
        g("pro");       // setResultPlanKey("pro") —— 纯前端展示，无后端调用
      }
    }, 5000);
  }catch{}
}, []);
```

#### 错误码体系（聊天 SSE error 事件）

| code | 含义 | action |
|------|------|--------|
| 429 | 限流 | - |
| 492 | 需付费升级 Pro | - |
| 493 | 需操作 | `set_default_browser` / `update_version` |

后端在用户发消息时检测 `unique-uuid` 标记位：非 `1` → 返回 493 + `action=set_default_browser`，前端弹「设为默认浏览器」提示。

#### 对 2api 项目的意义

**关键**：本项目请求时**必须带 `unique-uuid` header 且标记位 = `1`**，否则后端返回 493 拒绝服务。生成方式：

```ts
function makeProUuid(): string {
  const hex = '0123456789abcdef';
  const ts = Math.floor(Date.now()/1000).toString(16).padStart(8,'0').slice(-8);
  const tsPos = [2,7,11,14,18,21,25,28];
  const tsMap = new Map(tsPos.map((p,i) => [p, ts[i]]));
  let out = '';
  for (let i=0; i<32; i++) {
    if (i === 5) out += '1';                    // 标记位 = 已设默认浏览器
    else if (tsMap.has(i)) out += tsMap.get(i);
    else out += hex[Math.floor(Math.random()*16)];
  }
  return [out.slice(0,8),out.slice(8,12),out.slice(12,16),out.slice(16,20),out.slice(20,32)].join('-');
}
// headers['unique-uuid'] = makeProUuid();
// headers['x-req-ctx'] = Buffer.from('1.1.39.0').toString('base64');
```

⚠️ **合规警告**：伪造 `unique-uuid` 标记位绕过「设默认浏览器」要求，等同于冒充 Pro 用户，违反 Tabbit ToS，可能导致封号。仅供逆向研究理解机制，勿用于实际滥用。

## 5. SSE 响应协议

响应 `Content-Type: text/event-stream`，标准 SSE 格式：

```
event: message_start
data: {"...": "..."}

event: message_chunk
data: {"content": "你"}

event: message_chunk
data: {"content": "好"}

event: message_finish
data: {"...": "..."}

event: finish
data: {"model_name": "..."}

event: close
data: 
```

### 5.1 事件类型全表（7978 模块 `w()` 分发器）

| event | 含义 | 本项目处理 |
|-------|------|-----------|
| `ready` | 流就绪 | 忽略 |
| `title` | 会话标题 | 忽略 |
| `message_start` | 消息开始 | 标记开始 |
| `message_chunk` | **内容增量**（默认 event，无 `event:` 行时也按此处理） | **拼接 → OpenAI delta** |
| `message_finish` | 消息结束 | 标记结束 |
| `tool_start` / `tool_finish` | 工具调用 | 忽略（首期） |
| `rag_start` / `rag_finish` | RAG 检索 | 忽略 |
| `message_tool_calls` / `message_tool_call_delta` | 工具调用增量 | 忽略（首期） |
| `thinking` / `thinking_finished` | 推理过程 | 可选透传为 reasoning |
| `compress_start` / `compress_end` / `compressing` | 上下文压缩 | 忽略 |
| `switch_model` | 模型切换 | 忽略 |
| `browser_use_start` | Agent 浏览器自动化 | 忽略（Agent 不实现） |
| `usage` | token 用量 | 记录，转 OpenAI usage |
| `ttft` | 首 token 耗时 | 忽略 |
| `finish` | 整轮结束；若 data 含 `model_name` 则是并行模型分支结束 | 结束流 |
| `close` | 流关闭 | 关闭 SSE |
| `error` | 错误；data 含 `error/details/status/traceId` | 转 OpenAI error |
| `cancel` | 取消 | 结束流 |
| `risky` | 敏感词拦截 | 转错误 |

### 5.2 持久化事件（durable，Agent 场景）

`replay_start` / `replay_unavailable` / `run_queued` / `command_received` / `command_applied` / `command_rejected` / `run_finish` / `run_error` / `run_cancel` —— Agent 模式才有，本项目首期不涉及。

### 5.3 message_chunk data 结构（推断 + 实测确认）

```json
{
  "content": "增量文本",
  "message_id": "...",
  "model_name": "..."   // 首块可能带
}
```

> 实测时打印每个 chunk 的完整 JSON 确认字段名（可能还有 `role`、`index` 等）。

### 5.4 SSE 解析算法（原文 `x()` 函数）

```js
let reader = response.body.getReader();
let decoder = new TextDecoder();
let buffer = "", eventName = "", dataStr = "", eventId = "";
for(;;){
  let {done, value} = await reader.read();
  if(done) break;
  buffer += decoder.decode(value, {stream:true});
  let lines = buffer.split("\n");
  buffer = lines.pop() || "";          // 最后一段不完整，留到下次
  for(let line of lines){
    let trimmed = line.trim();
    if(trimmed === ""){
      // 空行 = 事件边界，派发 (eventName||"message_chunk", dataStr)
      if(dataStr) dispatch(eventName || "message_chunk", dataStr, eventId);
      eventName = ""; dataStr = ""; eventId = "";
      continue;
    }
    if(trimmed.startsWith("event: ")) eventName = trimmed.slice(7).trim();
    else if(trimmed.startsWith("data: ")) dataStr = dataStr ? dataStr+"\n"+trimmed.slice(6) : trimmed.slice(6);
    else if(trimmed.startsWith("id: ")) eventId = trimmed.slice(4).trim();
  }
}
```

`dispatch` → `JSON.parse(dataStr)` → 按 5.1 表分发。

### 5.5 错误处理

- 非 2xx：读 `response.text()`，解析 JSON 取 `detail` / `message` / `status_code`
- 网络中断：`onStreamInterrupted` 回调（可重连，首期直接报错）
- 499：刷新 sign-key 后重试一次

## 7. 待验证清单（实现时优先打通）

按优先级排序，每项都需要一次真实请求确认：

1. **登录态 Cookie 字段** —— 哪几个 cookie 是必须的（session? token?）

3. **chat_session_id 可空性** —— 能否随机生成一个让后端自动建会话；若不能，找创建端点
4. **message_chunk data 字段** —— `content` 之外还有什么
5. **GET 请求如何签名** —— body 为空时 `sha256("")` 的值固定，确认服务端接受
6. **签名 header 命名** —— 实测确认 `x-nonce`/`x-signature` 是否如逆向所示反直觉（先用逆向结果试，不行再交换）
7. **速率限制 / 配额** —— `/api/commerce/quota/v1/usage` 返回结构，超限时的错误码

### 探测脚本建议

实现第一步先写一个 `scripts/probe.ts`，用真实 Cookie 依次打：
- `GET /chat/sign-key`（确认 Cookie 有效 + 拿动态 key）
- `GET /proxy/v1/model_config/models?a=0&scene=chat`（确认签名通过 + 拿模型列表）
- `POST /api/v2/chat/completion`（最小 body，打印原始 SSE）

每步把请求/响应完整存到 `logs/` 供分析。

### 8.3 E() SSE 请求器（7978 模块，简化）

```js
function E(url, opts, callbacks, abortCtrl){
  let ac = abortCtrl || new AbortController();
  let headers = withTraceId({...opts.headers, Accept:"text/event-stream", "Cache-Control":"no-cache"});
  let traceId = headers["trace-id"];

  let doFetch = async () => {
    let sig = await sign(opts.body || "");        // y()
    return fetch(url, {...opts, signal:ac.signal, headers:{...headers, ...sig}});
  };

  return {
    promise: doFetch()
      .then(async e => 499===e.status && await refreshSignKey() ? doFetch() : e)
      .then(async e => {
        if(!e.ok){ /* 解析错误，callbacks.onError */ }
        return parseSSE(e, callbacks, ac.signal, traceId);  // x()
      })
      .catch(async e => { /* 网络错误 / abort */ }),
    abort: () => ac.abort()
  };
}
```

## 9. 风险与限制

| 风险 | 说明 | 缓解 |
|------|------|------|
| Cookie 过期 | 登录态失效需重新导出 | 提供配置项 + 健康检查端点 |
| 签名 key 变更 | 服务端更新 key，硬编码失效 | 实现 499 自动刷新；key 可配置 |
| 反爬 / 风控 | 高频请求可能触发 | 加请求间隔；遵守配额 |
| 协议变更 | Tabbit 版本更新可能改协议 | 锁版本；文档标注来源版本 1.1.39.0 |
| ToS 合规 | 逆向使用可能违反服务条款 | 仅供个人学习，勿商用/滥用 |
| Agent 路径不实现 | WebSocket `/api/agent/v2/ws` 复杂 | 首期只做 HTTP 聊天 |
